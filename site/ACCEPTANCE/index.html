    Acceptance¶

    This page defines the acceptance-oriented validation posture for the Fiddy prototype.

    Fiddy is intended to demonstrate that alcohol label artwork can be compared against expected application data using local OCR, deterministic validation rules, fuzzy matching, image-quality diagnostics, batch processing, performance monitoring, and reviewer-facing exports.

    Acceptance is based on runtime evidence generated by the application, not on unsupported claims.

    Acceptance Scope¶

    The prototype acceptance scope includes:

    • Label data extraction.
    • Application-versus-label comparison.
    • Government-warning validation.
    • Batch processing.
    • Five-second per-label performance monitoring.
    • Reviewer-facing output and downloads.
    • Accessibility controls.
    • Local-first security posture.
    • Azure-compatible container deployment.
    • No direct COLA integration.

    Acceptance Evidence¶

    Fiddy generates or supports the following acceptance evidence:

    Evidence Area Source
    OCR extraction src/ocr_engine.py and extracted-label records.
    Image quality src/visual_quality.py and image-quality notes.
    Field comparison src/label_rules.py and rule-result records.
    Government warning src/warning_validator.py and warning-validation results.
    Batch processing src/batch_processor.py and batch summary records.
    Performance src/performance_monitor.py and performance CSV exports.
    Output generation src/report_writer.py and application download controls.
    Accessibility src/accessibility_checklist.py and manual validation records.
    Acceptance status src/acceptance_checker.py and acceptance-summary records.
    Azure readiness Dockerfile, startup.sh, and docs/AZURE_DEPLOYMENT.md.

    Requirement Status Categories¶

    Acceptance checks should use the following status values:

    Status Meaning
    Met The requirement is implemented and supported by runtime or configuration evidence.
    Partially Met The requirement is implemented in part, but needs additional validation, documentation, or representative test evidence.
    Not Met The requirement is not implemented or current evidence shows the requirement failed.
    Not Evaluated The requirement cannot be evaluated from the current run or available evidence.

    Functional Acceptance Criteria¶

    Label Data Extraction¶

    Fiddy should extract or attempt to extract key alcohol label fields from uploaded label artwork.

    Expected fields include:

    • Brand name.
    • Class/type.
    • Alcohol content.
    • Net contents.
    • Producer, bottler, brewer, vintner, importer, or responsible party.
    • Country of origin when applicable.
    • Government warning text.

    Acceptance evidence includes OCR text, extracted-label fields, image-quality notes, and reviewer-facing extraction warnings.

    Application vs. Label Comparison¶

    Fiddy should compare application data against extracted or observed label evidence.

    The comparison output should include:

    • Field name.
    • Application value.
    • Extracted or observed value.
    • Status.
    • Severity.
    • Confidence.
    • Explanation.
    • Reviewer action.

    Fuzzy Matching¶

    Fiddy should use fuzzy matching where ordinary reviewer judgment can tolerate minor variation, such as casing, punctuation, spacing, or apostrophes.

    Fuzzy matching should not be used to pass exact government-warning text requirements.

    Government Warning Validation¶

    Fiddy should validate the government warning separately from ordinary fuzzy fields.

    The warning review should distinguish between:

    • Warning missing.
    • Warning present.
    • Exact text match.
    • Near match requiring review.
    • Visual formatting requiring human confirmation.

    Visual properties such as boldness, font size, contrast, placement, or hidden text may require human review.

    Batch Processing¶

    Fiddy should support manifest-driven batch processing.

    Acceptance evidence should include:

    • Manifest rows loaded.
    • Uploaded files loaded.
    • Matched files.
    • Missing files.
    • Extra files.
    • Processed files.
    • Skipped files.
    • Batch warnings.
    • Batch errors.
    • Per-label reports.

    Performance¶

    Fiddy should record per-label processing time and compare timing against the configured SLA.

    The default prototype target is:

    LABEL_PROCESSING_SLA_SECONDS = 5.0

    Performance evidence should include:

    • Processing seconds.
    • SLA seconds.
    • Within-SLA status.
    • SLA breach seconds.
    • Average seconds.
    • Median seconds.
    • P50 seconds.
    • P90 seconds.
    • P95 seconds.
    • SLA breach count.
    • SLA breach rate.
    • Performance acceptance status.

    Prototype Batch Size¶

    The prototype acceptance range is configured through:

    BATCH_ACCEPTANCE_MIN_FILES = 20
BATCH_ACCEPTANCE_MAX_FILES = 50

    Small smoke tests may be useful during development, but they should be marked as Not Evaluated for formal batch-size acceptance.

    Output and Downloads¶

    Fiddy should provide reviewer-facing outputs and downloadable files.

    Expected outputs include:

    • Summary CSV.
    • Detail CSV.
    • Comparison CSV.
    • Performance CSV.
    • JSON report.
    • Markdown report.

    Security and Data Handling Acceptance¶

    Fiddy should preserve a local-first prototype posture.

    Recommended prototype settings:

    Setting Recommended Value
    REQUIRE_LOCAL_OCR true
    ALLOW_EXTERNAL_ML_ENDPOINTS false
    ENABLE_UPLOAD_PERSISTENCE false
    ENABLE_RAW_TEXT_LOGGING false
    ENABLE_FILE_PATH_LOGGING false

    Exception logging should be sanitized and should not intentionally store:

    • Raw OCR text.
    • Manifest rows.
    • Uploaded file contents.
    • Full local file paths.
    • Secrets.
    • API keys.
    • Tokens.
    • Email addresses.

    Azure Acceptance¶

    Fiddy should be deployable as an Azure-compatible containerized workload.

    The container should include:

    • Python runtime.
    • Streamlit.
    • Tesseract OCR.
    • Poppler for PDF conversion.
    • Fiddy source code.
    • Local runtime configuration.

    The application should not require an external OCR endpoint or external machine-learning endpoint for prototype operation.

    Acceptance Validation Workflow¶

    Before stakeholder demonstration, run:

    python -m compileall app.py config.py booger.py src
mkdocs build

    Then run the application locally:

    streamlit run app.py

    For container validation:

    docker build -t fiddy:local .
docker run --rm -p 8501:8501 fiddy:local

    Manual Acceptance Checklist¶

    Use this checklist before demonstration:

    Check Expected Result
    App starts locally Streamlit opens without error.
    Manifest upload works CSV manifest loads and records are recognized.
    Label upload works Images, PDFs, or ZIP files are accepted.
    Verification completes Run Verification produces results.
    Summary appears Batch dashboard and summary table render.
    Comparison appears Side-by-side field comparison renders.
    Downloads work CSV, JSON, and Markdown downloads are available.
    Performance appears Per-label timing and SLA data are available.
    Simple Mode works Reviewer workflow remains upload, run, review/download.
    Advanced Mode works Diagnostics and detailed outputs are available.
    High Contrast works Interface remains readable.
    Large Text works Interface remains usable.
    Keyboard review works Main workflow can be reached through keyboard navigation.
    Logs are sanitized No raw OCR text, manifest rows, or full paths are written to logs.
    Container builds Docker image builds successfully.
    Container runs App opens from the running local container.

    Acceptance Notes¶

    Fiddy is a prototype reviewer-assist application. It does not make final compliance decisions, does not write results back to COLA, and does not replace human review for visually dependent determinations.
